# bpmnPJ 执行计划

## 1. 项目目标

本项目不是直接开发 Odoo 插件，而是建立一个独立的 `bpmn.js` 能力验证场，用来回答以下问题：

1. `bpmn.js` 生态里的核心插件能否稳定混装。
2. 哪些插件适合后续接入我们的审批/流程追踪场景。
3. 如何为未来的“流程追踪工作台”提供可复用的前端底座。
4. 如何为后续 Odoo 集成沉淀一套低风险的技术选型。

最终目标不是先做一个“全都要”的生产系统，而是做出一个：

- 可运行
- 可验证
- 可扩展
- 可回流到现有工作流模块

的 BPMN 前端实验项目。

---

## 2. 建设原则

### 2.1 技术原则

1. 独立项目优先，不与现有 Odoo 前端代码直接耦合。
2. 先验证底层能力，再接业务语义，不跳步骤。
3. 先追求稳定混装，再追求视觉复杂度。
4. 所有新增能力都必须有明确验收标准。
5. 所有第三方插件都要记录版本、兼容性、问题和替代方案。

### 2.2 范围原则

本项目当前阶段包含：

- BPMN 建模器能力验证
- BPMN 查看器能力验证
- 插件混装验证
- 业务属性扩展验证
- 流程追踪界面原型验证

本项目当前阶段不包含：

- 直接嵌入 Odoo 生产环境
- 多人实时协作
- 完整权限系统
- 后端流程执行引擎重写
- 复杂部署体系

## 5. 分阶段执行方案

## P0. 项目初始化

### 目标

建立一个最小、干净、可运行的前端工程，作为后续验证基座。

### 建议技术栈

- 构建工具：`Vite`
- 语言：原生 `JavaScript`
- 样式：原生 `CSS`
- 包管理：`npm`

### 原因

1. 先不用 React/Vue，避免框架层问题污染 BPMN 插件兼容性验证。
2. 先把复杂度集中在 `bpmn.js` 生态本身。
3. 后续需要迁移到 Odoo 时，原生 JS 更容易拆解和复用。

### 任务

1. 初始化 `package.json`
2. 安装 `vite`
3. 建立 `src/main.js`
4. 配置开发命令和构建命令
5. 建立基础页面骨架

### 验收标准

1. `npm install` 正常
2. `npm run dev` 可启动
3. 浏览器可打开空白项目首页

### 输出物

- 可运行的本地前端项目

---

## P1. 基础 Modeler 跑通

### 目标

完成最小 BPMN 建模器。

### 依赖

- `bpmn-js`

### 功能范围

1. 加载空白 BPMN 模板
2. 导入 BPMN XML
3. 导出 BPMN XML
4. 撤销/重做
5. 缩放/平移
6. 基础建模操作

### 任务

1. 封装 `createModeler.js`
2. 实现 `importXML`
3. 实现 `saveXML`
4. 实现 `saveSVG`
5. 实现简单工具栏
6. 实现空白模板加载

### 验收标准

1. 能新建、拖拽、连线 BPMN 元素
2. 能导入标准 BPMN 文件
3. 能导出修改后的 XML
4. 控制台无关键报错

### 输出物

- 最小建模器页面

---

## P2. 基础插件混装

### 目标

验证最常用、最稳妥的增强插件组合。

### 建议优先接入插件

1. `diagram-js-minimap`
2. `diagram-js-grid`
3. `bpmn-js-native-copy-paste`

### 任务

1. 建立 `modules.js`，统一管理插件注入
2. 按顺序逐个接入插件
3. 检查样式冲突
4. 检查缩放、拖拽、定位是否异常
5. 记录每个插件的兼容版本

### 验收标准

1. 小地图可用
2. 网格显示正常
3. 复制粘贴可用
4. 不破坏基础建模
5. 插件之间无明显 UI 冲突

### 输出物

- 第一版插件混装能力
- `docs/plugin-matrix.md` 初版

---

## P3. 属性面板与业务扩展

### 目标

建立“标准属性 + 自定义业务属性”的配置能力。

### 依赖

1. `bpmn-js-properties-panel`
2. 必要时接入 Camunda/Zeebe moddle 扩展

### 业务扩展方向

后续可在 BPMN 元素上承载以下属性：

1. 审批模式
2. 审批人配置
3. 抄送配置
4. 按钮绑定标识
5. 自动执行标识
6. 异常接管标识
7. Odoo 模型/动作映射

### 任务

1. 接入标准属性面板
2. 验证版本兼容性
3. 自定义 moddle 描述文件 `mt-approval.json`
4. 编写自定义 Provider
5. 在属性面板中渲染业务字段
6. 验证 XML 持久化与重载恢复

### 验收标准

1. 选中节点能编辑标准字段
2. 自定义字段可编辑
3. 导出 XML 后可保留业务扩展属性
4. 再次导入 XML 后字段不丢失

### 输出物

- 可编辑的业务 BPMN 模型

---

## P4. BPMN 校验与规则验证

### 目标

建立流程结构和建模规范校验能力。

### 依赖

1. `bpmnlint`
2. `bpmn-js-bpmnlint`

### 第一批规则建议

1. 必须存在开始事件
2. 必须存在结束事件
3. 用户任务必须命名
4. 网关必须具有有效出口
5. 避免悬空元素
6. 避免无标签连接线

### 任务

1. 建立 `bpmnlint.config.js`
2. 接入标准规则
3. 编写必要的自定义规则
4. 测试错误定位与提示样式

### 验收标准

1. 错误能在画布中标出
2. 错误列表能定位节点
3. 校验结果不影响正常建模

### 输出物

- 可实时提示错误的建模器

---

## P5. 执行模拟能力

### 目标

验证流程逻辑推演能力，为未来追踪工作台打基础。

### 依赖

- `bpmn-js-token-simulation`

### 任务

1. 接入模拟器
2. 验证顺序流、网关、并行结构
3. 验证与属性面板、小地图、Grid 的共存情况
4. 记录冲突和限制

### 验收标准

1. 能播放 Token 流转
2. 常见网关结构可视化正常
3. 不出现明显遮挡与事件冲突

### 输出物

- 可模拟执行路径的页面

---

## P6. 自动布局与导入修复

### 目标

解决“后端只给 XML 语义，不给图形坐标”的问题。

### 依赖

- `bpmn-auto-layout`

### 任务

1. 导入无 DI 的 BPMN 文件
2. 自动生成布局
3. 对比布局结果可读性
4. 评估是否适合放入后续实际业务链路

### 验收标准

1. 无布局 XML 可被正常展示
2. 自动布局后节点层次清晰
3. 导出的图能再次被导入

### 输出物

- 布局修复工具能力

## P7. Tracking 模式原型

### 目标

做一个接近我们现有“流程追踪工作台”方向的只读查看模式。

### 核心思想

Modeler 负责编辑，Tracking Viewer 负责展示运行态。

### Tracking 需要验证的能力

1. 节点状态高亮
2. 当前节点 pulse 效果
3. 异常节点高亮
4. 抄送/非阻塞节点弱化样式
5. 右侧轨迹时间线
6. 顶部实例状态摘要
7. 覆盖层显示处理人、耗时、异常信息

### 数据模型建议

前端 Tracking 模式先用 mock 数据，字段建议如下：

```json
{
  "instanceId": 1001,
  "state": "running",
  "currentNodes": ["Task_DeptApprove"],
  "completedNodes": ["StartEvent_1", "Task_Create"],
  "errorNodes": [],
  "pendingNonBlockingNodes": ["Task_CC"],
  "logs": [],
  "meta": {
    "title": "销售订单打印审批",
    "sourceDocument": "S00013",
    "triggerLabel": "申请打印"
  }
}
```

### 任务

1. 封装只读 Viewer
2. 把运行态数据映射为 marker
3. 做右侧轨迹栏
4. 做顶部状态条
5. 支持 mock 场景切换

### 验收标准

1. 同一份 BPMN 可在 Viewer 正常渲染
2. 各类状态 marker 可准确显示
3. 右侧轨迹和流程图状态一致

### 输出物

- Tracking 工作台原型页

---

## P8. Odoo 回流评估

### 目标

明确哪些能力值得回流到现有工作流模块，哪些能力应保留在独立项目中。

### 评估维度

1. 集成复杂度
2. 体积与性能成本
3. 与 Odoo assets 机制的兼容性
4. 是否需要服务端支撑
5. 是否适合审批业务场景
6. 是否具备长期维护价值

### 预期分类

#### 适合优先回流

1. Viewer 高亮能力
2. Tracking 工作台 UI
3. Overlay/Badge 信息展示
4. 自动布局能力
5. 轻量属性展示

#### 需要谨慎回流

1. 完整 Modeler
2. Token Simulation
3. 全量 Properties Panel
4. 大量第三方 Camunda 社区插件

### 输出物

- `docs/integration-notes.md`

## 7. 风险清单

## 7.1 版本兼容风险

### 问题

`bpmn-js`、`properties-panel`、`token-simulation`、`bpmnlint` 的版本可能不完全兼容。

### 应对

1. 先锁核心版本
2. 每次只引入一个插件
3. 建立插件兼容矩阵

## 7.2 CSS 冲突风险

### 问题

多个插件会引入自己的样式，容易出现按钮遮挡、图标冲突、层级异常。

### 应对

1. 所有项目样式做命名空间隔离
2. 插件样式单独管理
3. 对浮层和工具条统一做 z-index 管理

## 7.3 事件冲突风险

### 问题

建模器、模拟器、复制粘贴、属性面板都依赖事件总线，混装后容易相互影响。

### 应对

1. 插件逐个接入
2. 保留最小复现样例
3. 明确模块注入顺序

## 7.4 业务扩展过早风险

### 问题

过早加入审批业务字段，会掩盖底层兼容性问题。

### 应对

1. 前 3 个阶段只做基础能力
2. 业务字段放在属性面板稳定后再进入

## 8. 推荐推进顺序

建议严格按下面顺序推进，不要跳阶段：

1. P0 项目初始化
2. P1 基础 Modeler
3. P2 基础插件混装
4. P4 校验能力
5. P3 属性面板与业务扩展
6. P5 执行模拟
7. P6 自动布局
8. P7 Tracking 原型
9. P8 Odoo 回流评估

说明：

- `P4` 可以提前于 `P3`，因为 Lint 通常比属性面板更稳、更容易落地。
- `P5` 必须晚于基础插件稳定后再做。
- `P7` 应该在基础 Viewer、marker、overlay 能力清晰后再开始。

## 10. 成功标准

本项目成功，不是指“插件越多越好”，而是指：

1. 清楚知道哪些插件可以稳定混装
2. 清楚知道哪些能力适合我们审批业务
3. 清楚知道 Odoo 后续应该只接 Viewer 还是接更完整能力
4. 做出一套可验证、可演示、可复用的 BPMN 前端底座

---

## 11. 当前结论

当前最合理的推进方式是：

先把 `bpmnPJ` 做成一个独立的 BPMN 能力实验项目，
优先验证 `Modeler + 基础插件 + Lint + Tracking Viewer`，
等验证通过后，再决定哪些能力回流到现有 `MT_Approval_Workflow` 模块。
